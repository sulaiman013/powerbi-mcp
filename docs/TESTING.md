# Testing

The suites in `tests/` are **assert-based scripts** (no pytest required). Each prints
`[PASS]`/`[FAIL]` per check and exits non-zero on any failure. They run **without Power BI
or ADOMD** — pure logic is tested directly and live connectors are mocked.

## Run everything
```bash
python run_tests.py
```
Run one suite:
```bash
python tests/test_security_enforcement.py
```

## Engine-level verification of generated TMDL

Beyond the assert suites, the TMDL authoring output has been verified against Microsoft's own
engines (2026-07):

- **Parse:** a full project generated by `pbip_add_measures` + `pbip_create_date_table`
  (fiscal variant) + `pbip_add_hierarchy` + `pbip_add_calculation_group` was deserialized with
  **`Microsoft.AnalysisServices.Tabular.TmdlSerializer`** (the same AMO code path Power BI
  Desktop uses when opening a PBIP). Every property round-tripped: multi-line expressions,
  `///` descriptions, formats, folders, `dataCategory: Time` + `isKey`, sort-by wiring,
  hierarchy level order, calculation-group precedence/items/`formatStringDefinition`, and the
  Name/Ordinal companion columns.
- **Evaluate:** the same structures were created on a **live Power BI Desktop model** via TOM;
  the engine accepted them, a calculate refresh materialized the date table with exactly the
  expected row count and correct fiscal-year values, and calculation-group items executed in
  queries. All test objects were removed afterwards (verified via `INFO.TABLES()`/`INFO.MEASURES()`).

Note for live (TOM) workflows: a calculated table created through TOM needs a **calculate
refresh** to materialize rows; a PBIP-authored one materializes when Desktop opens the project.

## Coverage philosophy
- **Offline / pure logic** (security, PBIP rename, model analysis, diff, refresh
  classification, governance parsing) — fully unit/mock tested here.
- **Live paths** (Desktop/XMLA/TOM/REST/Admin) — orchestration is mock-tested and the API
  contracts are doc-verified against Microsoft Learn; **end-to-end verification needs a
  Windows + Power BI / Fabric environment** (see the live smoke-test checklist in the README).

## Adding a tool (keep the registry in sync)
A tool must be registered in three places in `src/server.py`: `handle_list_tools` (the `Tool`
spec), `_build_tool_dispatch` (the handler), and `_build_tool_annotations` (the safety hints).
A parity check (in the verification scripts) asserts all three contain the same names — keep
them aligned or `list_tools`/`call_tool` will drift.
